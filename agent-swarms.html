Swarms, and the economics
of who thinks.

Cursor's argument that a frontier planner steering a fleet of cheap workers beats one big agent — not because it's parallel, but because it keeps context clean. And that the cost of intelligence collapses when you only pay for it where decisions happen.

00 · Start here

If you've never heard of any of this

Skip this section if you already work with AI coding agents. Otherwise, everything below rests on five ideas:

  • Who wrote it. Cursor is a company that makes an AI-powered code editor. This is a post from their research team about their own system — so it's a vendor describing its own results (see the caveat at the end).
  • What an “agent” is. An AI model that doesn't just answer questions but does work in a loop — reads files, writes code, runs commands, checks results, tries again. Think of it as a tireless junior developer you give a task to.
  • What a “swarm” is. Instead of one agent working alone, you run hundreds at once on the same codebase. The obvious problem — and the subject of this article — is that they trip over each other constantly.
  • Frontier vs. cheap models. The smartest AI models (“frontier” models) cost a lot per unit of text processed. Weaker models are dramatically cheaper but make more mistakes. The whole economic question is which jobs actually need the expensive one.
  • Why cost is the story. AI models are billed per token — roughly a word-fragment of text read or written. A long agent run burns billions of them, so the same finished result can cost $1,000 or $10,000 depending purely on which model did which part.

The experiment in one sentence: they asked a swarm of AI agents to rebuild SQLite — a real, widely-used database program — from scratch in the Rust programming language, working only from the written manual, then measured how correct the result was and what it cost under different model combinations.

◆ Why that's a hard task

SQLite is not a toy. It's one of the most deployed pieces of software on earth — it runs inside browsers, phones, and operating systems — and its manual is 835 pages. Rebuilding it correctly is a genuine multi-engineer, multi-month project, which is exactly why it makes a serious test.

The one-liner

One expensive, smart AI (the planner) breaks the job into ever-smaller pieces; a crowd of cheap, fast AIs (the workers) build those pieces. Asked to rebuild the 835-page SQLite manual in Rust with no source code, no tests and no internet, every setup eventually passed 100% of a hidden correctness test. They differed only on price — $1,339 when a smart planner directed cheap workers, versus $10,565 when one strong model did everything. Same result, ~8× the price. And the reason it works isn't that many agents run at once — it's that splitting the roles keeps each AI's attention uncluttered.

01 · What a swarm actually is

Two roles, arranged as a tree

The system isn't “N copies of an agent racing.” Cursor's framing: a large task naturally takes the shape of a tree — a goal at the root that subdivides recursively into basic units of work. Two specialized roles walk that tree.

Depth is adaptive — “the swarm's shape grows to cover the problem's contours,” so compute and context scale with the task, not a fixed fan-out.

◆ Planner — frontier model
Holds the goal and the architecture. Subdivides recursively, makes the design decisions and trade-offs, delegates down — and never implements, so its context never fills with low-level detail.
● Workers — fast, cheap model
Each executes one narrow, delegated leaf and never plans — so it can spend its entire context on the single slice in front of it, uncluttered by anyone else's problem.

02 · The real mechanism

It wins on context, not parallelism

Key idea: an AI model has a context window — a limited amount of text it can hold in mind at once. Everything it's read, written, and been told competes for that space. Fill it with clutter and the model loses the thread. Think of it as a desk that only fits so much paper.

The headline surprise: the gains come mostly from context efficiency, not from doing many things at once. A single agent has to keep low-level implementation detail and high-level strategy on that one desk — so it drifts. Split the roles and the planner stays fixed on architecture while each worker's desk holds nothing but its own narrow job, uncluttered by anyone else's problem.

Framed another way: the swarm is a compiler for intent — the planner lowers a vague spec into progressively more concrete instructions, and infrastructure resolves the conflicts that scale inevitably produces.

Each AI jump raised the unit of abstraction

Cursor's ladder: every capability step let a human hand off a bigger chunk of intent. Swarms move the unit from a file to an entire specification.

03 · The experiment

Reimplement all of SQLite, in Rust, blind

The task: implement the entire 835-page SQLite manual in Rust. To force genuine construction rather than recall, the swarm was denied the source code, the test suites, the SQLite binary, and internet access. It had the spec and nothing else.

How it was graded

  • Hidden oracle. Runs were scored with sqllogictest — millions of queries with known-correct answers; the grade is the fraction the swarm's database gets right. The swarm was deliberately not told the suite existed, so it couldn't teach to the test.
  • Fixed budget. A 4-hour cutoff for every configuration.
  • Manual audit. After each run they hand-reviewed the code and the trajectory, checking for cheating and shortcuts and confirming the system was built out evenly — not faked at the edges to pass queries.

Four configurations

Same task, four model setups. GPT-5.5, Grok 4.5, Opus 4.8 and Fable 5 are frontier (expensive, capable) models from different AI labs; Composer 2.5 is Cursor's own fast, cheap model. “planner → worker” denotes a hybrid, where each role runs a different model; the other two run one model in both roles.

One term you'll need below: the harness is the surrounding machinery that organises the agents — how work is split, how commits are merged, how conflicts get resolved. It's not the AI model; it's the scaffolding around it. Cursor compares an old harness against their rebuilt new harness, holding the model constant, to isolate how much the scaffolding alone matters.

This is the result worth sitting with: every configuration on the new harness eventually passed 100% of the suite. The models did not separate on whether they could finish — they separated on what finishing cost. At the 4-hour cutoff the new runs sat at 73–85% and still climbing; the same swarm on the old harness ranged 11–77% and plateaued. Grok 4.5 makes the cleanest before/after: 80% in four hours on the new harness, versus an old-harness run that “spiraled” and had to be paused before hour two.

So the spread that matters is the price of the identical outcome — $1,339 to $10,565, an ~8× gap for the same 100%. That is what makes this an economics story rather than a capability one.

04 · The infrastructure problem

They had to build a new version-control system

At swarm scale you hit failure modes no human team produces. The key architectural choice: every change routes through a custom VCS — “so it is where collisions first become visible,” and several coordination mechanisms are implemented directly inside it.

Three terms first. Version control is the system that lets many people edit the same codebase without losing each other's work; Git is the near-universal tool for it. A commit is one saved change. A merge conflict is what happens when two parties edit the same lines and the system can't decide which version wins — a human normally resolves it by hand. Now imagine hundreds of agents generating those conflicts every second.

Git simply could not carry the load. Cursor's earlier browser swarm “peaked at roughly 1,000 commits per hour on Git”; the new system “peaks at around 1,000 commits per second” — a ~3,600× jump in raw throughput.

◆ Don't confuse these two numbers

Throughput went up ~3,600×, but the disciplined runs commit far less — ~968 in two hours versus the old run's 68,000. One is capacity (what the plumbing can absorb at peak), the other is behaviour (how much redundant thrash the coordination layer prevents). The new harness is built to survive a flood, and then arranges not to produce one.

The VCS is the choke point — so it's also where each failure mode below gets caught and resolved.

The article names four collapse modes at scale, each with the mechanism that answers it:

  • Split-brain — two planners independently build the same concept two different ways. → planners make design decisions themselves rather than delegating them, and must ensure no two subtrees decide the same question. Decisions live in shared design docs; dependent code carries a compile-checked reference back to its doc, so when planners contradict each other a reconciler merges the docs and the fix propagates downstream.
  • Merge collisions — workers constantly overwrite or abandon each other's edits on shared files. → a neutral third-party agent arbitrates each conflict on everyone's behalf, optimizing only for impartial, efficient resolution — like a merge queue.
  • Megafiles — hot files bloat and become the site of constant collisions. → workers can flag a bloated file; new commits are blocked and an outside agent decomposes it into smaller modules.
  • Ossification — agents stop daring to touch core code. → licensed intentional breakage: an agent judging a core change worthwhile makes a focused out-of-scope patch and leaves a comment explaining why; the compiler carries the change through, and everything depending on the old design fails to build — surfacing the work to whoever must fix it.

05 · The economics

Tokens flow one way, dollars the other

This is the “model economics” punchline. Workers do 69–90% of the tokens across configs — the sheer volume of work. But the dollars land elsewhere, because planner tokens (frontier model) cost far more each.

A tiny sliver of tokens carries most of the bill — because that sliver is where the thinking happens. Token shares are the reported range (“at least 69%, over 90% in most”); the planner’s ≈$928 is derived as $1,339 total − $411 workers, consistent with the article’s “roughly two-thirds.”

The clearest cut is the worker fleet in isolation. Doing essentially the same execution work, GPT-5.5's workers ran up $9,373; swap in cheap Composer 2.5 under an Opus planner and the entire worker fleet costs $411 — a ~23× drop on the bulk of the labor.

“Few moments in a large task genuinely require frontier intelligence — the original decomposition, the design decisions, and certain trade-offs.” — why the hybrid works: pay for the smart model only where it moves the needle

That's the whole optimization: concentrate frontier intelligence at the decision points, and push the enormous volume of execution onto a cheap model. You're not buying less intelligence — you're buying it only where it changes the outcome.

06 · Two things holding it together

Redundant review, and a shared field guide

  • Decorrelated review lenses. Reviewers are handed deliberately different slices — one gets the worker's full transcript, one only its output, one nothing but the codebase — and run on different models with different training and personality. No single lens catches everything, but decorrelated lenses stack the way self-driving systems reach above-human reliability with no single perfect component. The economics help: review is far cheaper than the work it audits, so the compute is high-return.
  • The field guide. A folder owned entirely by the agents, whose index.md is auto-injected into every agent at start; their only constraint is a line budget, so they curate it themselves. Because the AI models can't learn from experience mid-run (their internal “weights” are fixed once training ends), this shared note-file is the only way one agent's hard-won lesson reaches the next — so it's precisely the surprise encounters that are worth writing down, and the next agent's path gets shorter. Cursor names the principle: stigmergy, how ants and termites coordinate without talking — they shape the environment, and the environment shapes the next organism.

07 · How to hold this

The takeaway — and a caveat

The durable idea is clean and worth keeping regardless of the specifics: separate the agent that decides from the agents that do, keep each one's context narrow, and pay for frontier intelligence only at the decision points. Architecture (hierarchical decomposition) plus operational discipline (custom VCS, arbitration, stacked review) is what turns scale from a liability into a result.

⚠ Read the numbers as illustrative

This is a vendor post from Cursor, and the model names (GPT-5.5, Grok 4.5, Composer 2.5, plus Opus 4.8 / Fable 5) and the round figures read as forward-looking positioning rather than an audited, reproducible benchmark. Trust the shape of the argument — hybrid beats monolith, cost concentrates in the planner — more than any single dollar amount.

Glossary

Every term, in one line

Agent
An AI model that works in a loop — reads, writes code, runs it, checks, retries — rather than just answering once.
Swarm
Hundreds of agents working on the same codebase at the same time.
Planner / Worker
The two roles. The planner (smart, expensive) decides and splits the work; workers (cheap, fast) build the pieces.
Frontier model
A top-tier, expensive AI model. Here: GPT-5.5, Grok 4.5, Opus 4.8, Fable 5.
Composer 2.5
Cursor's own fast, cheap model, used as the worker in the hybrid runs.
Token
A word-fragment of text. AI models are billed per token, so token count = cost.
Context window
The limited amount of text a model can hold in mind at once — its “desk space.”
Harness
The scaffolding around the agents (how work is split, merged, and de-conflicted) — not the model itself.
SQLite
A real, extremely widely-used database program; the thing the swarm was told to rebuild.
Rust
The programming language they had to build it in.
sqllogictest
SQLite's own test suite — millions of queries with known-correct answers; the grade is the fraction the rebuild gets right.
Git / commit / merge conflict
Git = the standard tool for many people editing one codebase. A commit = one saved change. A merge conflict = two edits to the same lines that must be reconciled.
Crate
Rust's word for a self-contained code module.
