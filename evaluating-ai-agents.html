How Amazon grades its AI agents

A close read of Amazon's field guide to evaluating agentic AI — the framework, the metric taxonomy, and three production case studies. Rebuilt for someone who has never read the post and doesn't work in AI.

Source: "Evaluating AI agents: real-world lessons from building agentic systems at Amazon"
Authors: Yunfei Bai, Allie Colin, Kashif Imran, Winnie Xiong  ·  Published: Feb 18, 2026
This read: 2026-07-22

The 60-second version

  • An AI agent is an LLM that doesn't just answer — it plans, calls tools, remembers, and takes multi-step actions to finish a task. That extra machinery is exactly what old LLM tests can't see.
  • Amazon's answer is a two-part framework: a standard evaluation workflow (how you run an assessment) plus an evaluation library (what you measure), organized in three layers — the model, the components, and the whole system.
  • The metric list is broad on purpose: quality, task completion, tool use, memory, conversation, reasoning, and safety — because optimizing one and ignoring the rest produces an agent that looks good and fails in production.
  • Three real Amazon agents anchor it: a Shopping Assistant (onboarding thousands of tools), a Customer Service agent (routing intent), and a Seller Assistant (a multi-agent team that needs its coordination graded, not just each member).
  • The recurring lesson: automated metrics get you most of the way, humans close the gap, and you keep measuring in production forever — evaluation is a loop, not a launch gate.

01 Why grading an agent isn't grading a chatbot

Start here if "agent" and "LLM" sound like the same thing.

A large language model (LLM) — the technology behind ChatGPT, Claude, and Amazon's own models — is, at its core, a text predictor: give it a prompt, it returns words. You can grade one with a fixed test: ask a question, compare its answer to the right answer. That's a benchmark, and the field has thousands of them.

An AI agent wraps an LLM in the ability to act. Given a goal ("refund this order," "find me running shoes under $80"), it decides what steps to take, which external tools to call a "tool" = an API, database, or web service the agent can invoke, reads the results, remembers context, and loops until the job is done. The answer is no longer one sentence — it's a trajectory: a whole sequence of decisions and actions.

That breaks the old test. A benchmark scores the final sentence; it can't see that the agent picked the wrong tool, then got lucky, or reached a right-looking answer through broken reasoning that will fail on the next case. This is the article's central claim:

"While single-model benchmarks serve as a crucial foundation for assessing individual LLM performance in LLM-driven applications, agentic AI systems require a fundamental shift in evaluation methodologies." — verbatim, AWS ML Blog

So you have to grade the journey, not just the destination — and grade the pieces (tool choice, memory, planning) separately from the whole. That is what the rest of the framework is built to do.

02 The framework: a workflow plus a library

Two moving parts — one is the process, the other is the measuring kit.

Amazon splits the problem in two, described verbatim as "a generic evaluation workflow that standardizes assessment procedures across diverse agent implementations, and an agent evaluation library." Think of it as a recipe (the workflow — the same steps every team follows) and a pantry (the library — the reusable stock of metrics you pull from).

03 The evaluation library, in three layers

The "pantry" is stacked so you can zoom from the raw model up to the whole product.

The library evaluates at three altitudes. Grade only the top and you learn that the agent failed but not where; grade only the bottom and you miss whether the product actually helped anyone. You need all three.

04 What actually gets measured

The metric taxonomy — grouped by what part of the agent it interrogates. Quoted definitions look like this and are verbatim from the post; the rest are plain-English descriptions.

Final-response quality

Is the answer itself good?
Correctness
"The factual accuracy and correctness of an AI assistant's response to a given task."
Faithfulness
"Whether an AI assistant's response remains consistent with the conversation history."
Helpfulness
How well the reply actually addresses what was asked.
Relevance & conciseness
On-point to the question, and no padding.

Task completion

Did the job get done?
Goal success
"Did the AI assistant successfully complete all user goals within a conversation session."
Goal accuracy
Output compared against a known-correct answer (ground truth).

Tool use

Did it pick and call tools well?
Tool selection accuracy
"Did the AI assistant choose the appropriate tool for a given situation."
Tool parameter accuracy
Did it fill the tool's inputs with the right values?
Tool call error rate
How often tool invocations fail outright.
Multi-turn function-calling accuracy
Correct sequencing of tools across a conversation.

Memory

Does it recall the right context?
Context retrieval
Accuracy of what it pulls from memory, and whether it prioritizes the relevant bits.

Multi-turn conversation

Does it stay on-task over a chat?
Topic adherence
"assesses whether the conversation stays on predefined domains and topics."
Topic-adherence refusal
Does it correctly decline off-topic questions?

Reasoning & planning

Is the thinking sound, not just the answer?
Grounding accuracy
Do its understanding, tool choice and chain-of-thought line up?
Faithfulness / context score
Is each step logically consistent and grounded in context?

Responsibility & safety

Is it safe to ship?
Hallucination
"Do the outputs align with established knowledge, verifiable data, logical inference" — i.e. is it making things up?
Toxicity & harmfulness
Screening for offensive, hateful, violent or otherwise harmful output.

Cost & experience

The business dimensions
Cost
Compute/latency spend per task — a real agent has a budget.
Customer experience
The downstream impact on the person the agent serves.

Many of these are computed by an LLM-as-a-judge — a second model prompted to score the first agent's output — which is cheap and scalable but must itself be calibrated against human ratings (see Lesson 3).

05 Three agents, three hard problems

The framework earns its keep in production. Each case study surfaces a different failure mode — and a different fix.

Case 1 · Amazon Shopping Assistant

When your agent has thousands of tools

The problem
A shopping agent may need "hundreds, sometimes thousands, of tools from underlying Amazon systems" — search, pricing, reviews, inventory. Wiring each API in by hand "typically takes months to complete." And the more tools it has, the easier it is to pick the wrong one.
The fix
An LLM-powered self-onboarding system that auto-generates each tool's schema and description, plus cross-org standards for how tools are described — so the agent sees consistent, machine-readable tools. Regression tests run against "golden datasets" synthesized from historical logs.

Case 2 · Amazon Customer Service Assistant

Understanding what the customer actually wants

The problem
Intent detection — reading a customer's message and routing it to the right handler. Get the intent wrong and every downstream step is wrong, no matter how good the rest of the agent is.
The fix
An LLM-driven simulator that role-plays virtual customer personas, generating diverse, realistic scenarios (grounded in anonymized real interactions) to stress-test routing at scale — far more coverage than hand-written test cases.
Graded on
intent correctnessrouting accuracytask completiontopic adherence
Case 3 · Amazon Seller Assistant · multi-agent

Grading a team, not a soloist

The problem
This is a multi-agent system: a planner splits a request into subtasks and hands each to a specialist agent. Now you must grade not just each agent but the coordination between them — the part that has no equivalent in a single agent.
The fix — grade the coordination
On top of each agent's individual metrics, three inter-agent metrics: planning score (were subtasks assigned well?), communication score (efficient messaging between agents?), and collaboration success rate (% of subtasks completed). Human review catches emergent, edge-case behaviour the metrics miss.

06 The lessons, distilled

The four takeaways the authors want you to leave with.

1

Measure every dimension, not your favourite one

Quality, performance, responsibility and cost. Optimize one in isolation and you ship an agent that's accurate but slow, or fast but unsafe. Evaluation has to be holistic.

2

Standard metrics aren't enough — add domain metrics

Generic scores must be supplemented with application-specific success criteria, defined with domain experts, so "good" maps to a real business outcome rather than a benchmark number.

3

Humans in the loop are indispensable

Especially for high-stakes decisions. People audit reasoning chains, verify business alignment, produce the ground-truth labels for golden datasets, and calibrate the LLM-as-a-judge so the automated scores can be trusted.

4

Keep measuring in production, forever

Dashboards, alert thresholds and anomaly detection track real-world behaviour continuously; feedback loops trigger retraining and context tweaks. Evaluation is an operating loop, not a one-time launch gate.

⚠ Read this as a vendor source

This is an AWS Machine Learning Blog post, and it culminates in a pitch for Amazon Bedrock AgentCore Evaluations. That doesn't make it wrong — the taxonomy and the layered approach are genuinely useful — but trust the shape of the argument over any single figure.

The numbers are deliberately soft: "thousands" of agents, "hundreds, sometimes thousands" of tools, onboarding that takes "months". These are order-of-magnitude claims to establish scale, not measured statistics — none carry a baseline, a date range, or a methodology. Take them as "this is a big-company-scale problem," not as precise data.

08 Glossary

Every term of art used above, in one clause each.

Agentic AI / AI agent
An LLM given the ability to plan, call external tools, remember, and take multi-step actions toward a goal — not just answer a prompt.
LLM (large language model)
The text-prediction model at an agent's core (e.g. Claude, GPT, Amazon's Nova/Titan).
Foundation model
A large pre-trained model used as the base "engine" you build an agent on; picking it is the bottom layer of the library.
Trajectory
The full sequence of an agent's steps and decisions on a task — the thing you grade instead of just the final answer.
Trace file
A recorded log of one agent run (its steps, tool calls, outputs), captured offline or live, and fed into evaluation.
Tool / function calling
An agent invoking an external API, database, or service; "multi-turn function calling" means chaining several such calls correctly.
Intent detection
Working out what a user actually wants from their message, so the request can be routed correctly.
Topic adherence
Whether the agent stays within its allowed subject area and refuses off-topic requests.
Chain-of-thought (CoT)
The model's step-by-step intermediate reasoning; grading its coherence is part of reasoning evaluation.
Hallucination
Confident output that isn't grounded in real data or valid inference — i.e. the model making things up.
LLM-as-a-judge
Using a second LLM to score an agent's output automatically; scalable but must be calibrated against human ratings.
Ground truth
The known-correct answer a metric compares the agent's output against.
Golden dataset
A curated, trusted set of test cases (often synthesized from historical logs) used for regression testing.
Human-in-the-loop (HITL)
People inserted into the evaluation process to audit reasoning, label data, and calibrate automated judges.
Multi-agent system
Several specialist agents coordinated by a planner/orchestrator; needs its coordination graded, not just its members.
Regression testing
Re-running a fixed test set after a change to confirm nothing that used to work has broken.
