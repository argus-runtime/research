§1

What the article is trying to say

The whole page turns on a single shift in how you drive an agent.

Instead of steering an agent turn-by-turn, you declare the finish line once and let it self-correct until it crosses. You send a user.define_outcome event and the agent starts working immediately — no follow-up message required.

“An outcome tells the session what the end result should look like and how to measure its quality. The agent works toward that target, self-evaluating and iterating until the outcome is met.”

Defining “done”

Two parts: a plain-language description of the deliverable, plus a required rubric — a markdown doc of explicit, independently gradeable criteria.

Good: “The CSV contains a price column with numeric values.” Bad: “The data looks good.” Vague criteria produce noisy grades because each line is scored on its own.

How it checks itself

The harness auto-provisions a grader that scores the artifact against the rubric in a separate context window, so it isn’t swayed by the agent’s own reasoning.

The grader’s pass/fail explanation is handed back to the agent, which takes another pass. That feedback loop is the feature.

§2

The outcome loop

Define → work → grade → revise, until satisfied or out of iterations (default 3, max 20).

01
Define
Send description + rubric
02
Work
Agent builds the artifact
03
Grade
Separate grader scores each criterion
04
Revise
Feedback → next iteration
05
Deliver
Files land in the sandbox
↺ steps 02–04 repeat onneeds_revision

Every evaluation ends with a result that decides what happens next:

ResultWhat happens
satisfiedRubric met. Session goes idle.
needs_revisionAgent starts a new iteration cycle.
max_iterations_reachedOne final acknowledgment turn, then idle. No further grading.
failedRubric doesn’t apply to the deliverable (e.g. description and rubric contradict). Session goes idle.
interruptedA user.interrupt paused the current outcome so a new one can start.

Deliverables are written to /mnt/session/outputs/ inside the sandbox and pulled out via the Files API once the session is idle. Only one outcome runs at a time, but you can chain them in sequence.

§3

Where the environment fits in

Your honest question — and the honest answer.

The “Define outcomes” page barely covers the environment. It mentions it only in passing: a session is created against a pre-existing environment_id, and the agent runs in a sandbox that writes to /mnt/session/outputs/. The article’s focus is outcomes, graders, and rubrics — not the environment.

The environment’s importance is really its own page (managed-agents/environments), which you asked me to fold in. Here’s why it matters: the environment is the world the agent acts in — get it wrong and even a perfect rubric can’t be satisfied.

  • what it isA reusable cloud sandbox config. Create once, reference its ID per session. Many sessions share one environment, but each gets a fresh, isolated Linux container — no shared filesystem state.
  • packagesPre-installs tools before the agent starts (apt, cargo, gem, go, npm, pip), cached across sessions. If a rubric needs pandas or ffmpeg, this is where it comes from.
  • networkingunrestricted (default, minus a safety blocklist) or limited to an allowed_hosts list. Docs recommend limited + least privilege for production.
  • runtimesCommon languages, databases and utilities ship pre-installed in every cloud sandbox.
  • lifecycleEnvironments persist until archived or deleted. Not versioned — keep your own record of changes so you know which config a session ran under.

The link to outcomes: the outcome says what “done” means; the environment decides whether the agent has the tools, packages, and network reach to get there. A rubric that expects an .xlsx built with a library the sandbox never installed will simply keep failing.
